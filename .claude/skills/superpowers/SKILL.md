# Superpowers - 独立开发者核心命令集

## Overview

提供结构化的项目规划和执行工作流，从创意到落地的完整流程支持。

## Commands

### /brainstorm - 头脑风暴

用于项目初期创意探索和需求分析。

**触发词**: "brainstorm", "头脑风暴", "想法", "创意"

**流程**:
1. 收集用户输入的核心想法
2. 扩展相关功能点
3. 识别技术挑战
4. 提出多种实现方案
5. 输出结构化的创意文档

**输出格式**:
```markdown
# 项目头脑风暴: [项目名]

## 核心想法
- 主要功能点
- 用户价值

## 功能扩展
- 必要功能 (MVP)
- 增值功能 (Nice-to-have)
- 未来规划

## 技术考量
- 技术栈选择
- 潜在挑战
- 依赖关系

## 实现方案
### 方案 A: [名称]
- 优点:
- 缺点:
- 工作量:

### 方案 B: [名称]
- 优点:
- 缺点:
- 工作量:

## 推荐方案
[选择理由]
```

### /write-plan - 制定计划

将头脑风暴结果转化为可执行的实施计划。

**触发词**: "write-plan", "制定计划", "写计划", "规划"

**流程**:
1. 分析项目需求
2. 拆解任务模块
3. 定义依赖关系
4. 排列优先级
5. 输出实施计划

**输出格式**:
```markdown
# 实施计划: [项目名]

## 项目概述
- 目标:
- 范围:
- 技术栈:

## 里程碑

### Phase 1: 基础搭建
- [ ] 任务 1.1: [描述]
- [ ] 任务 1.2: [描述]

### Phase 2: 核心功能
- [ ] 任务 2.1: [描述]
- [ ] 任务 2.2: [描述]

### Phase 3: 优化上线
- [ ] 任务 3.1: [描述]
- [ ] 任务 3.2: [描述]

## 依赖关系
- 任务 2.1 依赖 1.1
- 任务 3.1 依赖 2.1, 2.2

## 风险评估
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| [风险1] | 高/中/低 | [措施] |

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
```

### /execute-plan - 执行计划

逐步执行计划中的任务。

**触发词**: "execute", "执行", "开始做", "实施"

**流程**:
1. 读取当前计划状态
2. 识别下一个可执行任务
3. 执行任务
4. 更新任务状态
5. 报告进度

**执行原则**:
- 每次只执行一个任务
- 任务完成后立即标记
- 遇到阻塞时报告并跳过
- 保持计划文档同步更新

### /status - 查看状态

查看当前项目进度和待办事项。

**触发词**: "status", "状态", "进度", "现在到哪了"

**输出**:
```markdown
## 项目状态: [项目名]

### 进度统计
- 总任务: X
- 已完成: X (XX%)
- 进行中: X
- 待开始: X

### 当前任务
[当前正在执行的任务描述]

### 下一步
[接下来要做的任务]

### 阻塞项
[如果有阻塞，列出]
```

---

### /review - 复盘总结

项目或阶段完成后的复盘。

**触发词**: "review", "复盘", "总结", "回顾"

**输出**:
```markdown
## 项目复盘: [项目名]

### 完成情况
- 计划任务: X
- 实际完成: X
- 完成率: XX%

### 亮点
- [做得好的地方]

### 改进点
- [可以改进的地方]

### 经验教训
- [学到的经验]

### 后续行动
- [需要跟进的事项]
```

## 工作流示例

```
用户: 我想做一个食谱分享网站
Claude: [使用 /brainstorm 展开讨论]

用户: 方案A不错，帮我制定计划
Claude: [使用 /write-plan 生成实施计划]

用户: 开始执行吧
Claude: [使用 /execute-plan 逐步实施]

用户: 现在进度如何？
Claude: [使用 /status 汇报进度]
```

## Best Practices

1. **先规划后执行**: 不要跳过 brainstorm 和 plan 阶段
2. **小步快跑**: 任务粒度要小，便于追踪
3. **及时同步**: 每完成一步立即更新状态
4. **灵活调整**: 计划可以根据实际情况调整
